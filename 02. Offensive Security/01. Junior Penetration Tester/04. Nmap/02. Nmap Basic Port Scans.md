### What is Port Scanning?
Port scanning checks which ``ports on a host are open``, ``closed``, or ``filtered``.

Example:
````
Target: 10.10.10.10

Port 22  → Open   (SSH)
Port 80  → Open   (HTTP)
Port 443 → Closed (HTTPS)
````
Open ports usually mean a service is listening.

## Task 2: TCP and UDP Ports
A **port** identifies a specific **network service** running on a computer.

<img width="1200" height="958" alt="image" src="https://github.com/user-attachments/assets/ee8b5954-49bc-4bff-91a9-66fcba031e70" />


Think of a computer as a house and ports as its doors.
- **IP Address** = House address
- **Port** = Door
- **Service** = Person or application behind the door


### Common Ports
| Port    | Service | Protocol |
| ------- | ------- | -------- |
| **22**  | SSH     | TCP      |
| **80**  | HTTP    | TCP      |
| **443** | HTTPS   | TCP      |
| **53**  | DNS     | TCP/UDP  |
| **21**  | FTP     | TCP      |


### Port States

#### 1. Open Port : service is listening

<img width="650" height="800" alt="image" src="https://github.com/user-attachments/assets/070ebf6d-c24c-4adc-83f1-507ccbe60ba1" />

Example:
- SSH running on port 22
- Web server running on port 80

#### 2. Closed Port | No Service
No service is listening.

<img width="1086" height="1448" alt="image" src="https://github.com/user-attachments/assets/663912cd-07ce-411d-8996-ca283401859b" />

The port exists but nothing is using it.



#### 3. Filtered Port | Firewall Blocking
A firewall blocks the packets.

<img width="1086" height="1448" alt="image" src="https://github.com/user-attachments/assets/3db762a1-0082-40a9-a2fe-662fb85e2d7f" />


Nmap **cannot determine** whether the port is open or closed.



#### 4. Unfiltered Port
The port is reachable, but Nmap cannot determine whether it is open or closed.

This state is commonly seen during a **TCP ACK scan** (``-sA``).


#### 5. Blocked Port
A blocked port means we cannot reach the port because a firewall, router, or security device is preventing access.

#### 6. Open | Filtered
Nmap cannot determine whether the port is ``open`` or ``filtered``.

This is commonly seen in UDP scans, where an open UDP port often sends no response.



#### 7. Closed | Filtered
Nmap cannot determine whether the port is closed or filtered because it receives insufficient information.

#### Nmap Port States

| **Port State**      | **Meaning**                                                                               | **Accessible?** | **Service Running?** | **Common Cause**                                            |
| ------------------- | ----------------------------------------------------------------------------------------- | --------------- | -------------------- | ----------------------------------------------------------- |
| **Open**            | A service is listening on the port.                                                       | ✅ Yes           | ✅ Yes                | SSH, HTTP, HTTPS, DNS, etc.                                 |
| **Closed**          | No service is listening, but the port is reachable.                                       | ✅ Yes           | ❌ No                 | Service is stopped or not installed.                        |
| **Filtered**        | Nmap **cannot determine** whether the port is open or closed because packets are blocked. | ❌ No            | Unknown              | Firewall, ACL, IDS/IPS blocks traffic.                      |
| **Unfiltered**      | Nmap can reach the port but **cannot determine** whether it is open or closed.            | ✅ Yes           | Unknown              | Usually seen with **TCP ACK (`-sA`)** scans.                |
| **Open|Filtered**   | Nmap cannot determine whether the port is open or filtered.                               | Unknown         | Unknown              | Common in **UDP scans** or Firewall silently drops packets. |
| **Closed|Filtered** | Nmap cannot determine whether the port is closed or filtered.                             | Unknown         | Unknown              | Insufficient response from the target.                      |

## Task 3: TCP Flags
TCP flags are ``1-bit values`` in the TCP header that control how a TCP connection is **started**, **managed**, and **closed**.



### TCP Header (Focus on Flags)
<img width="2400" height="1480" alt="image" src="https://github.com/user-attachments/assets/f4db426a-27e9-49b4-81f1-db34ff358dd0" />


### TCP Flags Summary
| **Flag** | **Full Name**  | **Purpose**                                         | **Example**                      |
| -------- | -------------- | --------------------------------------------------- | -------------------------------- |
| **URG**  | Urgent         | Data is urgent and should be processed immediately. | Rarely used today                |
| **ACK**  | Acknowledgment | Confirms data has been received.                    | Every established TCP connection |
| **PSH**  | Push           | Send data immediately to the application.           | Interactive applications (SSH)   |
| **RST**  | Reset          | Immediately terminates or rejects a connection.     | Port is closed                   |
| **SYN**  | Synchronize    | Starts a TCP connection (3-way handshake).          | Opening a connection             |
| **FIN**  | Finish         | Gracefully closes a TCP connection.                 | Ending a connection              |


#### 1. SYN (Start Connection)
Used to start a TCP connection.
````
Client
   │
SYN
   ▼
Server
````
**Remember**: SYN = **Start**


#### 2. SYN + ACK
Server accepts the connection.
````
Client
   │
SYN
   ▼
Server
   │
SYN + ACK
   ▲
````
Means:

> I received your SYN and I'm ready.


#### 3. ACK
Client confirms the connection.
````
Client
   │
ACK
   ▼
Server
````
Now the TCP connection is established.

TCP 3-Way Handshake
````
Client                     Server
   │                         │
   ├──── SYN ───────────────►│
   │◄── SYN + ACK ───────────┤
   ├──── ACK ───────────────►│
   │                         │
Connection Established
````


#### 4. PSH
Sends data directly to the application.
````
Client
   │
PSH + ACK
   ▼
Server
````
Example:
- SSH typing
- Telnet
- Interactive applications


#### 5. RST
Immediately resets the connection.
````
Client
   │
SYN
   ▼
Server
   │
RST
   ▲
````
Usually means:
- Port is closed
- Connection rejected


#### 6. FIN
Gracefully closes the connection.
````
Client
   │
FIN
   ▼
Server
````
Means:
> I'm finished sending data.


#### 7. URG
Marks urgent data.
````
URG = Process this data immediately
````
Rarely used today.

## Task 4: TCP Connect Scan (-sT)

A **TCP Connect Scan** (``-sT``) checks whether a TCP port is open by completing the entire TCP 3-way handshake.
<img width="1200" height="589" alt="image" src="https://github.com/user-attachments/assets/e8cb41ee-3d30-42e4-bb39-e60613e2f7d7" />


It is the default TCP scan for users without ``root/sudo privileges``.


### How TCP Connect Scan Works
#### If the port is Open
````
Attacker                      Target
    │                           │
    ├──── SYN ────────────────► │
    │ ◄── SYN, ACK ────────────┤
    ├──── ACK ────────────────► │
    │                           │
Connection Established
    │
    ├──── RST, ACK ───────────► │
    │
Connection Closed
````
Explanation
1. Attacker sends SYN.
2. Target replies SYN/ACK.
3. Attacker sends ACK.
4. Connection is established.
5. Nmap immediately sends RST/ACK to close the connection because it only wants to know if the port is open.


#### If the port is Closed
````
Attacker                      Target
    │                           │
    ├──── SYN ────────────────► │
    │ ◄──── RST, ACK ──────────┤
````
Explanation
- Target immediately rejects the connection.
- Nmap marks the port as Closed.



#### Basic Command
````
nmap -sT TARGET
````

## Task 5: TCP SYN Scan (-sS)
A **TCP SYN Scan** (``-sS``) is the default Nmap scan when run as root/sudo. It is called a Half-Open or Stealth Scan because it does not complete the TCP 3-way handshake.


### How TCP SYN Scan Works
#### Open port:
````
Attacker                      Target
    │                           │
    ├──── SYN ────────────────► │
    │ ◄── SYN, ACK ────────────┤
    ├──── RST ────────────────► │
````
Explanation
1. Send SYN
2. Target replies SYN/ACK
3. Nmap immediately sends RST
4. Connection is never established

Result:
````
Port = Open
````


#### Closed Port
````
Attacker                      Target
    │                           │
    ├──── SYN ────────────────► │
    │ ◄──── RST, ACK ──────────┤
````
Result:
````
Port = Closed
````

## Task 6: UDP Scan

### UDP Open Port
<img width="1024" height="800" alt="image" src="https://github.com/user-attachments/assets/6a7627b5-0501-4af4-9ce9-ee7c3ae72f4e" />

What happens?
- Nmap sends a UDP packet to the target.
- The UDP port is OPEN.
- Many UDP services (DNS, SNMP, NTP, etc.) do not reply to an empty UDP packet.
- Because there is no ICMP error, Nmap assumes the port is open (or open|filtered in some cases).

### UDP Closed Port


What happens?

Nmap sends a UDP packet.

The port is closed, so the operating system replies:
````
ICMP Type 3
Code 3
Destination Unreachable
Port Unreachable
````
This tells Nmap:

> Nothing is listening on this UDP port.

Therefore Nmap marks it as:
````
closed
````

## Task 8: Summary

### 1. Scan Types
| Scan Type            | Command                      | Description                                                     |
| -------------------- | ---------------------------- | --------------------------------------------------------------- |
| **TCP Connect Scan** | `nmap -sT 10.48.168.23`      | Completes the full TCP 3-way handshake. No root required.       |
| **TCP SYN Scan**     | `sudo nmap -sS 10.48.168.23` | Half-open (stealth) scan. Requires root/sudo.                   |
| **UDP Scan**         | `sudo nmap -sU 10.48.168.23` | Scans UDP ports. Closed ports reply with ICMP Port Unreachable. |

### 2. Useful Nmap Options
| Option                  | Purpose                                        | Example                             |
| ----------------------- | ---------------------------------------------- | ----------------------------------- |
| `-p-`                   | Scan all **65,535** ports                      | `nmap -p- TARGET`                   |
| `-p1-1023`              | Scan ports **1–1023**                          | `nmap -p1-1023 TARGET`              |
| `-F`                    | Scan the **100 most common** ports             | `nmap -F TARGET`                    |
| `-r`                    | Scan ports in **consecutive order**            | `nmap -r TARGET`                    |
| `-T0` ~ `-T5`           | Timing templates (`T0` slowest → `T5` fastest) | `nmap -T4 TARGET`                   |
| `--max-rate 50`         | Send **≤ 50 packets/sec**                      | `nmap --max-rate 50 TARGET`         |
| `--min-rate 15`         | Send **≥ 15 packets/sec**                      | `nmap --min-rate 15 TARGET`         |
| `--min-parallelism 100` | Run **at least 100 probes in parallel**        | `nmap --min-parallelism 100 TARGET` |
